Visión del proyecto

Desarrollar una App de Citas dentro del Backoffice Homie, como nuevo módulo operativo de la plataforma, para gestionar de punta a punta el agendamiento de visitas a propiedades en arriendo en Chile.

La iniciativa busca digitalizar un proceso que hoy tiene una alta carga operativa y manual, centralizando la configuración, coordinación, ejecución y seguimiento de citas dentro de una plataforma única, trazable y escalable.

El proyecto se estructura en cuatro capas complementarias: Admin/Backoffice (motor operativo), User Facing Booking Experience (cara al inquilino), Advisor Experience (operación en campo) y Communications (comunicaciones automatizadas vía Darwin.ia). Aunque el desarrollo inicial es para Chile, la arquitectura se diseña desde el inicio como base escalable para otros mercados y futuros módulos operativos de Homie.

Problema
El funnel se rompe antes de la primera visita

Homie Rent Chile no tiene hoy un flujo digital de booking. Esto genera alta carga operativa manual, baja visibilidad sobre disponibilidad de asesores e inmuebles, poca trazabilidad del estado de las citas y dificultad para confirmar, reprogramar o reasignar. El resultado es una experiencia inconsistente para el inquilino, el asesor y el propietario.

Los datos del funnel de arrendatarios de la semana 16 de 2026 (datos del reporte semanal disponible — se actualizará con datos 2026) muestran que la tasa de conversión Lead → Cita agendada es del 7.7%. Sin embargo, una vez que la visita ocurre, la tasa de conversión Cita → Reserva es del 74%. El cuello de botella no es la propiedad ni el precio: es la fricción para agendar.

Qué es esta capa y por qué existe

El Backoffice Homie ya contiene el REM — el módulo de inventario y gestión de propiedades. Esta capa agrega el Módulo de Booking como una nueva capacidad del backoffice, al mismo nivel que el REM, dentro de la misma plataforma operativa.

Sin esta capa el sistema de citas no tiene motor: no hay forma de definir quién puede mostrar qué propiedad, cuándo está disponible, ni de gestionar el estado de cada visita. Es la infraestructura sobre la que se apoyan todas las demás capas.

Esta capa está pensada como la primera expansión del Backoffice Homie más allá del REM. La sección de Usuarios es una capacidad transversal que luego podrán usar otros módulos. Los patrones de roles y permisos se diseñan para escalar a medida que el backoffice incorpore nuevos módulos.

Definición de negocio — Usuarios y roles

El sistema debe incorporar una sección de Usuarios a nivel global del admin. No es una necesidad puntual del módulo de booking — es una capacidad estructural sobre la cual se apoyarán otros módulos. En esta fase, la definición funcional de roles y permisos estará orientada principalmente al equipo de booking.

◎
Principio: role-based experience, no solo role-based access
El rol no solo define qué puede hacer el usuario — define cómo se presenta la experiencia. Un admin ve navegación completa y capacidades de configuración. Un asesor ve una experiencia mobile-first orientada a ejecución diaria. Un operador ve foco en seguimiento. Esta distinción debe estar presente en el diseño desde el inicio, aunque los roles estén hardcodeados en esta fase.

Definición de negocio — Disponibilidad de asesores

El sistema debe permitir definir la disponibilidad operativa base de cada asesor mediante una configuración semanal estructurada. Esta configuración es la regla principal sobre la cual se construye la agenda disponible del asesor y sirve como base para el cálculo de slots agendables.

La lógica no debe modelarse como disponibilidad genérica o textual, sino como una configuración explícita compuesta por selección de días de la semana, rangos horarios dentro de cada día y timezone asociada. Esto es importante porque el sistema opera hoy en Chile pero debe quedar preparado para múltiples países.

EspecificaciónEstructura de disponibilidad base
Días laborables
Selección múltiple de días de la semana. Si un día no tiene bloques configurados → no laborable.
Rangos horarios
Uno o más bloques por día (ej: Lunes 09:00–13:00 / 14:00–18:00). Los bloques no pueden solaparse dentro del mismo día.
Timezone
Explícita por configuración. Default: America/Santiago. Escalable a America/Mexico_City y otros. Nunca depender del timezone del servidor.
Reutilización
La disponibilidad base es una "semana tipo" persistente. No se reconfigura cada semana manualmente.
Jerarquía de disponibilidad — regla de prioridad

Cuando el sistema calcula si un slot es agendable, aplica estas tres capas en orden de prioridad. Una capa superior siempre invalida las inferiores.

1
Exclusiones globales · prioridad máxima
Feriados, cierres operativos, eventos internos. Invalidan cualquier disponibilidad. Si existe una exclusión global, ningún asesor puede ser agendado en esa fecha/hora. Al crear o editar una exclusión, el sistema detecta citas existentes impactadas y exige resolución antes de guardar.
2
Excepciones individuales
Vacaciones, licencias, permisos, aperturas extraordinarias de agenda. Aplican solo al asesor específico en las fechas/horas definidas. Sobreescriben la disponibilidad base de ese asesor.
3
Disponibilidad base del asesor
Semana tipo configurada. Solo válida si no existe excepción individual ni exclusión global que la invalide para esa fecha y hora específica.
  • ✓
    Configuración semanal base por asesor
    Interfaz de selección de días + rangos horarios múltiples por día. Timezone explícita. Validación de solapamientos. Persiste como semana tipo reutilizable.
  • ✓
    Excepciones individuales
    Bloqueos o aperturas extraordinarias sobre fechas específicas. Tipos: vacaciones, licencia, permiso, apertura extra. Día completo o franja horaria. Prioridad 2 en la jerarquía.
  • ✓
    Exclusiones globales
    Feriados, cierres operativos. Afectan a todos los asesores. Timezone explícita. Al crear/editar sobre fechas con citas existentes: el sistema lista las citas impactadas y bloquea hasta que sean resueltas.
Definición de negocio — Vinculación inmueble-asesor

El sistema debe permitir vincular asesores a inmuebles para que el módulo de booking pueda determinar quiénes están habilitados para mostrar cada propiedad. Un slot solo tiene sentido si existe una combinación válida entre un inmueble habilitado, al menos un asesor asignado, una ventana horaria disponible y ausencia de bloqueos.

  • ✓
    Asignación asesor a inmueble (1:1 en esta fase)
    Cada inmueble tiene un único asesor asignado. Un inmueble sin asesor no es agendable. La arquitectura del modelo de datos debe diseñarse para soportar 1:N en el futuro sin refactor conceptual.
  • ✓
    Indicador de estado de configuración del inmueble
    El listado de inmuebles muestra si cada uno está "listo para booking" o no. Condiciones: activo + asesor asignado + booking habilitado + sin restricciones bloqueantes. Filtrable en el listado.
  • ✓
    Cálculo de slots agendables
    Endpoint que devuelve los slots disponibles para una propiedad en una fecha dada, aplicando la jerarquía completa: disponibilidad del asesor → excepciones → exclusiones globales → citas ya tomadas. Resultado: slots con status available / grouped / unavailable.
Definición de negocio — Gestión de citas

El admin debe contar con una sección central de Gestión de Citas desde la cual el equipo pueda visualizar y operar todas las citas vinculadas al proceso de booking. Esta sección funciona como el centro operativo del módulo y está pensada para trabajo diario, no solo para consulta histórica.

User Facing Booking Experience
Experiencia del inquilino en la ficha de propiedad. Sin rediseñar la ficha existente — solo se agregan CTAs inteligentes según el estado de disponibilidad y se habilita el flujo de agendamiento completo desde la misma página.
Qué es esta capa y por qué existe

Esta capa habilita al inquilino para agendar una visita directamente desde la ficha de la propiedad, sin salir de la página ni crear una cuenta. Es la capa que convierte el tráfico existente en citas — y donde está la mayor oportunidad de mejora según los datos del funnel.

El objetivo no es rediseñar la ficha de propiedad ni modificar la estructura del journey existente. Se agregan y condicionan CTAs inteligentes según el estado de disponibilidad de cada propiedad. Si hay disponibilidad configurada, el usuario puede agendar. Si no la hay, se le ofrece contactar por WhatsApp.

Definición de negocio — UX / Diseño

La experiencia debe ser de baja fricción, alta claridad y visibilidad inmediata de la disponibilidad. El usuario no debe tener que hacer click previo para descubrir si hay o no horarios disponibles. La disponibilidad debe estar visible directamente en la página de propiedad.

Principios de UX
Disponibilidad
Visible directamente en la ficha sin click previo. Si hay disponibilidad, el selector está expuesto. Si no hay, el CTA "Contactar" está expuesto. El estado es inmediatamente legible al llegar a la página.
Contexto
El usuario no abandona la ficha en ningún paso del flujo. Todo ocurre en modales o microestados embebidos dentro de la misma página.
CTA principal
Estado disabled hasta que día + horario estén seleccionados. El cambio de disabled → enabled debe ser visualmente claro y notorio.
Visitas grupales
Horarios con citas existentes se muestran con indicación visual diferenciada y texto "Este horario ya tiene visitas agendadas — si continúas, la visita será grupal." El usuario puede continuar, pero debe entender el contexto antes del formulario.
Mobile
El flujo completo debe funcionar en mobile con la misma lógica. Selector de horarios como chips apilables. Modal como pantalla completa en mobile.
Features — Flujo A: con disponibilidad configurada
  • ✓
    Disponibilidad visible en página sin click previo
    Cuando la propiedad tiene disponibilidad configurada, el selector de días se muestra directamente. Ningún día ni horario viene preseleccionado por defecto. El CTA principal está en estado disabled.
  • ✓
    Selector de días y horarios con tres estados
    Disponible (estado normal) · Disponible grupal (indicador visual diferenciado) · No disponible (deshabilitado). Al seleccionar un día, los horarios de ese día aparecen sin redirección ni scroll mayor.
  • ✓
    Formulario modal con 4 campos y T&C obligatorios
    Se abre sobre la ficha al hacer click en el CTA activo. Header del modal muestra el resumen de la selección. Campos: Nombre, Apellido, Correo, Teléfono con selector de código de área. Términos y condiciones explícitos (no opt-in opcional) con justificación de por qué se piden los datos: "para coordinar la visita, enviar el detalle del agendamiento y gestionar cambios".
  • ✓
    Validación OTP vía WhatsApp con sesión persistente
    Al enviar el formulario, se envía un código por WhatsApp al número ingresado. El usuario lo ingresa en el modal. Si ya validó en la misma sesión y navegador (dentro de las últimas 24h de actividad), este paso se omite y se muestra "Teléfono validado". Al expirar por inactividad: "Tu validación expiró por inactividad. Vuelve a confirmar tu teléfono."
  • ✓
    Pantalla de éxito (congrats) con contexto completo
    Reemplaza el modal al completarse el agendamiento. Muestra: ícono o estado visual de confirmación, nombre de la propiedad, dirección completa, día y hora de la visita, nombre del arrendatario. Incluye dos CTAs: "Ver más propiedades" (link al portal de Homie) y "Agregar al calendario" (genera evento en Google Calendar o iCal con los datos de la visita). El copy debe dejar claro si la visita quedó agendada o pendiente de confirmación por parte del equipo.
Features — Flujo B: sin disponibilidad
  • ✓
    CTA "Contactar" con mensaje predefinido a WhatsApp
    Cuando la propiedad no tiene disponibilidad o no está configurada, se muestra directamente el CTA "Contactar". Al hacer click, abre WhatsApp con mensaje predefinido: "Hola, estoy interesado/a en la propiedad [URL de la propiedad]. ¿Pueden darme más información?". El usuario no necesita escribir nada. El link de la propiedad da contexto al asesor.

Qué es esta capa y por qué existe

El asesor es quien ejecuta la visita física. Hoy no tiene una herramienta digital que le dé el contexto necesario para llegar preparado, contactar al inquilino fácilmente y registrar el resultado de la visita. La experiencia del advisor no debe copiar la del admin general. Es una interfaz diferenciada por rol, con navegación simplificada y componentes pensados para uso con una sola mano, en la calle, posiblemente con poco tiempo disponible.

Definición de negocio — UX / Diseño

La app debe responder siempre a estas cuatro preguntas: ¿qué tengo que hacer ahora? → ¿qué información necesito para hacerlo? → ¿qué acción rápida debo ejecutar? → ¿cómo cierro esta tarea? Una acción principal por momento. Contexto mínimo, ejecución máxima.

Principios de UX
Navegación
Bottom navigation (comportamiento nativo de app). Tres tabs: Home · Agenda · Visitas. No replicar la navegación del admin general.
Tap targets
Mínimo 44px de altura en todos los elementos interactivos. Los CTAs principales deben ser prominentes y alcanzables con el pulgar.
Formularios
Bottom sheet o modal compacto. El formulario post-visita es la única excepción que puede requerir más espacio (5 preguntas), pero debe sentirse corto y rápido.
Visitas grupales (1:N)
Card agrupada por propiedad + hora. Dirección y horario una sola vez. Indicador de N personas. Expandible para ver cada contacto y sus CTAs individuales. Una sola ejecución física, múltiples contactos.

Features — Bloque "Antes de salir" (datos de acceso)

Dentro del detalle de la visita debe existir un bloque específico de datos de acceso a la propiedad. Este bloque muestra información operativa crítica para ejecutar correctamente la visita. Los datos provienen del REM a nivel propiedad.

Especificación del bloque
Contenido mínimo
Status llaves · Código de candado · Clave de candado. Preparado para agregar más campos de acceso en iteraciones futuras.
Dato faltante
Si algún dato no existe, mostrar explícitamente "Sin información" — nunca dejar vacío de forma ambigua. El asesor debe saber que falta, no asumir que está en otro lugar.
Tratamiento visual
Section card con fondo diferenciado del resto del detalle. Alta visibilidad. Label + value pairs. Status chip para el estado de llaves. Fácil lectura en mobile con poca luz.
Observaciones
Campo de texto para observaciones del inmueble: acceso especial, mascotas, restricciones. Recordatorio de llegada anticipada si aplica.
Iteración futura
Botón "Copiar" para código y clave — no obligatorio en esta fase pero la estructura del componente debe soportarlo.
Features — Formulario post-visita obligatorio

Cuando el asesor marca una visita como Concretada, el sistema exige completar un formulario obligatorio antes de cerrar definitivamente la tarea. La acción "Marcar como realizada" no cierra la visita directamente — abre el formulario como paso intermedio. Sin completarlo, la cita no pasa a estado Concretada.

Protocolo de escalamiento a humano

Darwin solo envía mensajes estructurados del flujo. Cualquier interacción fuera de los botones previstos activa el escalamiento. El arrendatario nunca debe sentir que está atascado con un bot.

💬
Texto libre fuera del flujo
Cualquier mensaje que no sea respuesta a un botón previsto. Preguntas sobre precio, documentos, dirección, mascotas, etc.
"En un momento te comunicamos con un asesor que puede ayudarte." · Notifica al asesor con conversación completa.
🙋
Solicitud de hablar con una persona
"Quiero hablar con alguien", "¿hay un humano?", "¿me pueden llamar?", expresión de frustración.
Escalamiento inmediato de alta prioridad. Alerta prioritaria al asesor.
🔄
Segundo intento de reprogramación fallido
El arrendatario intentó reprogramar pero la URL no tiene disponibilidad o falló técnicamente y vuelve a escribir.
Escalamiento con contexto "sin disponibilidad para reprogramar". El asesor busca horario alternativo manualmente.
✅
Intención de arriendo post-visita
El asesor contacta al arrendatario directamente vía el mensaje de Wapisimo y el arrendatario responde con intención de avanzar.
Conversación gestionada directamente por el asesor. No pasa por Darwin.
◎
Mensaje de puente obligatorio al escalar
Antes de pasar el hilo, Darwin siempre envía: "En breve te contactará un asesor de Homie que puede ayudarte con eso." El arrendatario nunca queda en silencio esperando sin saber qué pasa.

Communications
Recuperación automática de no-show

Cuando el asesor registra un no-show, Darwin envía automáticamente un mensaje al inquilino 2–4 horas después del horario de la visita. El mensaje ofrece reagendar con un link directo a la disponibilidad actualizada de la propiedad.

Hola [Nombre], vimos que no pudiste ir a ver [Propiedad] hoy. La propiedad sigue disponible. Si quieres reagendar, puedes elegir otro horario aquí: [link]
Reagendar visita
Por qué está fuera de esta fase
Requiere que el registro de no-show por parte del asesor sea consistente antes de automatizar mensajes sobre él. En etapas tempranas, enviar este mensaje sobre un registro incorrecto genera una experiencia negativa. Primero establecer el hábito de registro del asesor.
Backoffice · visibilidad operativa
Vista por lead — historial completo de visitas

Una vista en el backoffice que permite ver, por cada lead o inquilino, todas las propiedades que visitó, cuáles confirmó y a cuáles no fue, el resultado del formulario post-visita del asesor en cada caso, y el estado actual del proceso. Permite al equipo de operaciones tener una visión completa del comportamiento del inquilino en lugar de solo ver citas aisladas.

Información que debería incluir la vista por lead
→Datos de contacto del lead: nombre, teléfono, correo
→Lista de propiedades visitadas con fecha, asesor y resultado (Concretada / No-show)
→Respuesta del formulario post-visita del asesor por cada propiedad visitada
→Respuesta al mensaje de cierre post-visita (interesado / explorar más / sin respuesta)
→Estado actual del lead en el proceso de renta
Por qué está fuera de esta fase
Requiere datos acumulados de múltiples visitas por lead para ser útil. En la fase inicial la mayoría de los leads tendrá pocas visitas registradas. Es más valioso cuando el sistema lleva meses operando.
Vista para el propietario — visibilidad sobre sus visitas

Una interfaz o reporte para que el propietario pueda ver las visitas programadas a su propiedad: quién va, cuándo, y cuál fue el resultado. Hoy el propietario no tiene visibilidad directa de este proceso.

Por qué está fuera de esta fase
Requiere definir el canal y formato de comunicación con el propietario. Decisión que involucra al equipo de negocio. Se deja como mejora una vez que el flujo interno esté estabilizado.
Admin / Backoffice
Vinculación 1:N asesor-inmueble

Permitir que un mismo inmueble tenga más de un asesor asignado, habilitando mayor cobertura operativa. La arquitectura de datos se diseña para soportarlo desde el inicio, pero la UI y la lógica de asignación se limitan a 1:1 en esta fase.

Por qué está fuera de esta fase
Agrega complejidad en la lógica de asignación de citas. No es necesario para la operación inicial en Chile. La arquitectura lo soporta sin refactor.
Advisor Experience
Analytics personales del asesor

Métricas de rendimiento dentro de la app del asesor: visitas completadas, tasa de no-show, tasa de interés post-visita. Útil para motivación y seguimiento de performance, pero requiere datos acumulados para ser significativo.

Modelo distribuido de showing (red de asesores)

Cualquier persona habilitada por Homie puede mostrar propiedades dentro de una red operativa distribuida, similar a un modelo tipo Uber. La arquitectura de la Capa 3 se diseña con este modelo en mente para que la expansión sea natural.

Link al hilo de Darwin desde la app del asesor

Un acceso directo dentro del detalle de visita que lleva al asesor a la conversación de Darwin con ese arrendatario, permitiéndole ver el estado del flujo de confirmación sin salir de su app. Útil para entender si el arrendatario confirmó, reprogramó o no respondió, sin necesidad de consultar otro sistema.

Por qué está fuera de esta fase
Requiere integración entre Darwin y la app del asesor para obtener el link de la conversación por número de teléfono. Depende de que la API de Darwin exponga este endpoint. Bajo impacto en la operación inicial.
